## 摘要
代码生产现已成为一种商品；瓶颈在于确定构建内容并证明其有效性。我们提出了 Kitchen Loop，这是一个基于统一信任模型的自主、自演化软件框架：(1) 规范面（specification surface），列举了产品声称支持的功能；(2) “作为用户 x 1000”，即 LLM 代理以 1000 倍于人类的速度，作为合成高级用户对该规范面进行演练；(3) 无懈可击的测试（Unbeatable Tests），即代码作者无法伪造的基准真相验证；以及 (4) 漂移控制（Drift Control），具有自动暂停门的持续质量测量。我们在两个生产系统中进行了超过 285 次迭代验证，生成了 1,094 个以上的合并拉取请求，且回归预测器（regression oracle）未检测到任何回归（方法见第 6.1 节）。我们在大规模应用中观察到了涌现属性：多迭代自纠正链、自主基础设施修复以及单调改进的质量门。这些原语并非首创；我们的贡献在于将它们组合成一个经过生产测试的系统，并具备使长期自主演化变得安全的运行规程。

## Abstract
Code production is now a commodity; the bottleneck is knowing what to build and proving it works. We present the Kitchen Loop, a framework for autonomous, self-evolving software built on a unified trust model: (1) a specification surface enumerating what the product claims to support; (2) 'As a User x 1000', where an LLM agent exercises that surface as a synthetic power user at 1,000x human cadence; (3) Unbeatable Tests, ground-truth verification the code author cannot fake; and (4) Drift Control, continuous quality measurement with automated pause gates. We validate across two production systems over 285+ iterations, producing 1,094+ merged pull requests with zero regressions detected by the regression oracle (methodology in Section 6.1). We observe emergent properties at scale: multi-iteration self-correction chains, autonomous infrastructure healing, and monotonically improving quality gates. The primitives are not new; our contribution is their composition into a production-tested system with the operational discipline that makes long-running autonomous evolution safe.

### 1. 核心问题与整体含义
*   **研究动机**：随着大语言模型（LLM）的普及，代码生成已成为“廉价商品”。然而，当前的瓶颈在于**“知道该构建什么”**以及**“证明构建的内容确实有效”**。
*   **背景**：现有的 AI 编程助手（如 Cursor, Devin）虽然能提高短期速度，但往往导致代码复杂性增加、质量下降（即“Vibe Coding”现象）。论文旨在解决如何让 AI 代理在没有人类持续监督的情况下，安全地、系统化地维护和扩展复杂的生产级代码库。

### 2. 方法论
Kitchen Loop 提出了一个**统一信任模型**，通过六个阶段的循环实现自演化：
*   **规范面（Specification Surface）**：定义产品声称支持的所有功能矩阵（功能 × 平台 × 动作）。
*   **作为用户 x 1000 (AaU1000)**：LLM 代理模拟高级用户，以 1000 倍于人类的速度遍历规范面，寻找功能缺失或错误。
*   **不可伪造的测试（Unbeatable Tests）**：采用四层验证架构（编译、执行、解析、状态增量）。核心是验证“状态增量”（如账户余额变化），这是 AI 无法通过修改测试脚本来伪造的真实结果。
*   **对抗性 UAT 门禁**：实现“信息防火墙”，由一个不了解实现细节的“弱模型”担任测试员，严格按照测试卡执行验证，防止 AI 代理自我作弊。
*   **漂移控制（Drift Control）**：通过回归预言机（Regression Oracle）持续监测质量趋势，设有自动暂停门（Pause Gates），一旦检测到质量下降或异常，立即停止循环。
*   **讨论管理器（Discussion Manager）**：引入多模型（Gemini, GPT, Claude）辩论机制，处理架构决策和复杂评审，减少单一模型的偏见。

### 6. 主要结论与发现
*   **零回归**：在 1,000 多个合并的 PR 中，回归预言机未检测到任何由循环引入的错误。
*   **自主修复**：系统展现出“涌现属性”，能够自主发现并修复自身基础设施的 Bug（如内存分配错误、API 语法错误）。
*   **质量单调提升**：通过持续的测试和修复，产品的规格覆盖率和测试通过率稳步上升。
*   **范式转移**：证明了在具备严谨验证机制的前提下，软件开发可以从“人工编写代码”转向“人工定义规范与验证逻辑”。

### 7. 优点
*   **系统性强**：不只是简单的代码生成，而是构建了一个包含需求发现、执行、验证、评审的完整闭环。
*   **防作弊机制**：对抗性 UAT 和不可伪造测试解决了 LLM 在自动化测试中容易“走捷径”的顽疾。
*   **高性价比**：极大地降低了软件维护和回归测试的单位成本。

### 8. 不足与局限
*   **预言机依赖**：系统的安全性高度依赖于“回归预言机”的质量。如果预言机本身存在盲点，系统可能会引入无法察觉的错误。
*   **规范定义门槛**：要求产品必须有可枚举的规范面。对于需求极其模糊或处于早期探索阶段的项目，该框架难以启动。
*   **单线程限制**：目前的编排器是单线程运行的，虽然吞吐量已远超人类，但在超大规模并行开发上的表现尚未验证。
*   **领域局限**：目前仅在 DeFi 和数据处理领域验证，对于涉及复杂 UI/UX 审美判断或高度主观性需求的领域，其适用性受限。
