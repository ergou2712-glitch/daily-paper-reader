## 摘要
支撑基于图的检索增强生成（Graph-RAG）的知识图谱（KGs）越来越多地通过大语言模型（LLM）驱动的提取自动构建，而非由专家人工维护。正确的评估需要对所有相关阶段（提取、图构建和推理）进行检测，并保持足够的连贯性以定位故障，从而避免某一阶段的错误直到最后才被发现为错误答案。我们提出了 TRIAGE，这是一个针对自动化的、基于文档的 Graph-RAG 的阶段感知检测框架，它不仅询问底层图是否可信，还询问查询该图的成本。TRIAGE 为三个阶段附加了特定于阶段且可独立解释的指标：KG 实现（三元组置信度、源覆盖率以及模式和规范化检查）、专家 KG 验证（图级结构质量，其中正确性和完整性仅在有参考资料时作为离线校准计算）以及 KG 使用（检索覆盖率、忠实度和检索成本）；部署的指标不需要金标准标注，需要金标准的指标仅用于离线校准。在使用时，这些指标形成了一个必要条件的诊断链，其第一个断裂环节可定位故障，且诊断结果对应于可以补救故障的阶段杠杆：提取、图与模式或检索。TRIAGE 是一个理论框架，包含概念验证和可重复的评估协议。

## Abstract
Knowledge graphs (KGs) that underpin Graph-based Retrieval-Augmented Generation (Graph-RAG) are increasingly built automatically by LLM-driven extraction rather than curated by experts. Proper evaluation would require instrumenting all pertinent stages: extraction, graph construction, and inference, coherently enough to localize failures, so that a failure at one stage is not discovered as a wrong answer at the end. We introduce TRIAGE, a stage-aware instrumentation framework for automated, document-grounded graph-RAG that asks not only whether the underlying graph can be trusted but at what cost it can be queried. TRIAGE attaches stage-specific, independently interpretable metrics to three stages: the KG Implementation (triple confidence, source coverage, and schema and canonicalization checks), the KG Validation by expert (graph-level structural quality, with correctness and completeness computed only as offline calibration when a reference is available), and the KG Usage (retrieval coverage, faithfulness, and retrieval cost); the deployed metrics need no gold annotations, the gold-requiring ones serving only as offline calibration. At usage time these metrics form a diagnostic chain of necessary conditions whose first broken link localizes the failure, and the diagnosis maps to the stage levers that can remedy it: extraction, graph and schema, or retrieval. TRIAGE is a theoretical framework with a proof of concept and a reproducible evaluation protocol.

*   **核心问题**：如何量化评估自动构建的 KG 是否可信？如何在不依赖“金标准（Gold Standard）”标注的情况下，在推理阶段实时定位故障发生的具体环节（是提取、图构建还是检索出了问题）？

### 2. 方法论：核心思想与关键技术
TRIAGE 将 Graph-RAG 流程划分为三个阶段，并为每个阶段设计了特定的指标：

*   **核心思想**：构建一个**诊断链（Diagnostic Chain）**。在推理时，指标按顺序排列（如：查询是否在谱内 -> 实体是否被检索 -> 路径是否连通）。第一个断裂的环节即为故障点。
*   **关键技术细节**：
    *   **阶段 1：KG 实现（Implementation）**：关注三元组提取质量。
        *   *关键指标*：**TCS（三元组置信度）**，利用 LLM 输出 token 的几何平均概率作为置信度代理；**SCR（源覆盖率）**，衡量文档被成功提取的比例。
    *   **阶段 2：KG 验证（Validation）**：关注图的全局结构。
        *   *关键指标*：**DNR（死节点率）**，识别度数 $\le 1$ 的孤立节点；**LC（逻辑一致性）**，通过本体推理检测冲突。
    *   **阶段 3：KG 使用（Usage）**：关注检索与推理。
        *   *关键指标*：**ERC（实体检索覆盖率）**，衡量查询实体是否进入检索子图；**RRS（推理就绪得分）**，衡量检索到的实体间是否存在连通路径；**RPC（检索路径成本）**，衡量检索的计算开销。
*   **修复映射（Remediation Map）**：诊断结果直接指向修复杠杆。例如，低 RRS 指向“图与模式杠杆”（需改进实体消歧），而低 SCR 指向“提取杠杆”（需调整提示词或分块策略）。

### 6. 主要结论与发现
*   **诊断有效性**：结构化指标（ERC, RRS）与证据覆盖率（CC）高度相关。当 ERC=1 时，证据覆盖率接近 98.4%，而 ERC 低时则大幅下降。
*   **沉默成功风险**：实验发现即使检索结构不完整，强力 LLM 仍有约 80% 的概率答对（依赖参数知识）。这证明了**在生成答案前独立评估检索质量**的必要性，否则系统在知识更新时会产生不可控的幻觉。
*   **成本可见性**：通过 RPC 指标，框架首次让“检索覆盖率 vs 计算成本”的权衡变得可观测。

### 7. 优点
*   **阶段感知**：不再只看最终答案，而是能精准定位是提取、图结构还是检索的锅。
*   **无需金标准**：核心指标在部署阶段无需参考答案即可计算，具有极强的实操性。
*   **预防性分流**：可以在 LLM 生成答案前拦截低质量检索，节省算力并减少幻觉。

### 8. 不足与局限
*   **依赖 Log-probs**：TCS 指标依赖 LLM 输出的 token 概率，这在某些闭源 API 中难以获取。
*   **实体提取误差**：指标的准确性高度依赖于初始的实体识别（NER）和链接效果，若第一步识别错了，后续指标会产生连锁偏差。
*   **修复验证尚浅**：虽然提出了修复杠杆映射，但尚未在大规模生产环境中验证这些修复措施的成本效益比。
*   **本体依赖**：部分高质量指标（如 SC, LC）仍需要预定义的本体（Ontology），在完全无模式的开放域场景下效果受限。
