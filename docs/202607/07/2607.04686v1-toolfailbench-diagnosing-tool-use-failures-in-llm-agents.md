## 摘要
工具调用是现代语言模型智能体的核心，但综合基准测试分数往往掩盖了工具使用的具体失败点。一个从不调用所需工具的模型与一个调用了工具但忽略结果的模型，在最终任务准确率上可能表现相似。我们推出了 ToolFailBench，这是一个诊断性基准测试，用于衡量金融、医学、法律、网络安全和房地产等领域 1,000 个任务中的工具使用故障。需要工具的任务会返回模型无法猜测的数值，迫使其信任工具；而对照任务虽然附带了相同的工具，但应直接回答。我们使用规则分类器和两个通过多数投票汇总的大语言模型评判员，将每个轨迹标记为“跳过工具”（Tool-Skip）、“忽略结果”（Result-Ignore）、“输出伪造”（Output-Fabrication）和“不必要的工具使用”（Unnecessary-Tool-Use）。在 19 个主流模型中，表现最好的模型达到了 86.33% 的“纯净工具使用率”（Clean Tool-Use Rate），表明忠实的工具使用尚未达到饱和。更重要的是，综合得分相似的模型失败方式各异：大多数模型在无需工具的对照任务中保持克制，而 Llama-3.1 模型表现出“总是调用”（Always-Call）模式；在相同参数规模下，Llama-3.1-70B 和 Qwen2.5-72B 在对照任务准确率上相差 89 个百分点。工具使用评估不仅应衡量智能体是否调用了工具，还应衡量其是否正确使用了工具输出，以及在不需要时是否避免使用工具。

## Abstract
Tool calling is central to modern language model agents, but aggregate benchmark scores often hide where tool use fails. A model that never calls a needed tool and a model that calls the tool but ignores the result can look similar under final task accuracy. We introduce ToolFailBench, a diagnostic benchmark for measuring tool-use failures across 1,000 tasks in finance, medicine, law, cybersecurity, and real estate. Tool-required tasks return values the model wouldn't guess, forcing it to trust the tool while control tasks attach the same tools but should be answered directly. We label each trace with Tool-Skip, Result-Ignore, Output-Fabrication, and Unnecessary-Tool-Use, using a rule classifier and two LLM judges aggregated by majority vote. Across 19 headline models, the best reaches 86.33% Clean Tool-Use Rate, showing that faithful tool use is not saturated. More importantly, models with similar aggregate scores fail in different ways: most stay disciplined on no-tool controls, while Llama-3.1 models show an Always-Call pattern, and at the same parameter scale Llama-3.1-70B and Qwen2.5-72B differ by 89 percentage points on control-task accuracy. Tool-use evaluation should measure not only whether agents call tools, but whether they use tool outputs correctly and avoid tools when none is needed.

### 1. 核心问题与整体含义
*   **研究动机**：当前的 LLM 工具使用基准测试（如 BFCL, ToolLLM）通常只关注最终的成功率或准确率。这种“黑盒”评估掩盖了模型失败的具体原因：模型是因为根本没调用工具而失败，还是调用了工具却忽略了结果，亦或是凭空捏造了信息？
*   **核心问题**：如何系统性地诊断和区分 LLM 智能体在工具使用循环中不同阶段的故障模式？
*   **整体含义**：论文提出了 **ToolFailBench**，一个包含 1,000 个任务的诊断性基准测试，旨在通过细粒度的分类法揭示模型在工具调用决策、结果整合及输出忠实度方面的具体缺陷。

### 2. 方法论
*   **核心思想**：利用“参数化陷阱”（Parametric Traps）强制模型在自身记忆与工具返回结果之间做出选择。
*   **关键技术细节**：
    *   **任务设计**：包含 750 个“必须使用工具”的任务和 250 个“无需工具”的对照任务。在必须使用工具的任务中，模拟工具返回的值会故意与模型预训练中可能学到的常识或历史数据（先验知识）相矛盾。
    *   **故障分类法（Taxonomy）**：
        *   **Tool-Skip (TS)**：应调用而未调用工具。
        *   **Result-Ignore (RI)**：调用了工具但最终答案忽略了工具返回的证据，转而使用先验知识。
        *   **Output-Fabrication (OF)**：调用了工具但在答案中捏造了工具未提供的结构化信息。
        *   **Unnecessary-Tool-Use (UTU)**：在对照任务中不必要地调用工具。
    *   **评估流程**：采用“规则分类器 + 两个独立 LLM 评判员（Qwen 和 GLM）”的集成方案，通过多数投票确定最终标签，以平衡规则的确定性和语义理解的灵活性。

### 3. 实验设计
*   **数据集/场景**：涵盖五个专业领域：金融、医学、法律、网络安全和房地产。每个领域 200 个任务。
*   **对比方法**：评估了 19 个主流模型，包括：
    *   **闭源模型**：GPT-5.4-Mini, Claude-Sonnet-4.5, Claude-Haiku-4.5, Grok-4.3, Grok-4-1-Fast-Reasoning, DeepSeek-V4-Flash。
    *   **开源/推理模型**：Qwen2.5/3.5/3.6 系列、Llama-3.1 系列、Gemma-4 系列、GLM-4.7-Flash、QwQ-32B。
*   **Benchmark 指标**：Clean Tool-Use Rate (CTUR, 纯净工具使用率)、TSR (跳过率)、RIR (忽略率)、OFR (伪造率)、UTR (不必要调用率) 和 CTRL-Acc (对照任务准确率)。

### 4. 资源与算力
*   **硬件使用**：
    *   **推理阶段**：使用 H200 和 H100 GPU 进行模型推理。
    *   **评判阶段**：使用 B200 GPU 运行 LLM 评判员。

### 5. 实验数量与充分性
*   **实验规模**：共 1,000 个任务，对 19 个模型进行了全量测试，生成了数万条评估轨迹。
*   **充分性与客观性**：
    *   **多维度评估**：不仅有领域划分，还有任务类型（必须 vs 无需）划分。
    *   **一致性校验**：计算了评判员之间的一致性（Fleiss’ κ 为 0.693），并进行了“同家族偏见”检查（证明 Qwen 评判员并未偏袒 Qwen 模型）。
    *   **统计显著性**：为核心指标提供了 95% 的 Wilson 得分置信区间。实验设计较为客观、公平。

### 6. 主要结论与发现
*   **性能未饱和**：即使是表现最好的模型（Grok-4.3），其 CTUR 也仅为 86.33%，说明忠实使用工具仍有提升空间。
*   **失败模式差异显著**：总分相近的模型失败原因迥异。例如，Llama-3.1 系列表现出明显的“总是调用”（Always-Call）倾向，在对照任务中 UTR 极高（>77%）。
*   **规模不代表纪律**：在相同参数规模下，Qwen2.5-72B 在对照任务上的准确率比 Llama-3.1-70B 高出 89 个百分点，说明工具使用的“纪律性”更多取决于模型家族和训练行为，而非单纯的参数量。
*   **领域敏感性**：金融领域最容易出现“忽略结果”的情况，可能是因为该领域的先验知识（如股票代码关联）对模型的干扰最强。

### 7. 优点
*   **诊断深度**：超越了单一的成功率，提供了“为什么失败”的详细分解。
*   **陷阱设计**：通过参数化陷阱有效区分了模型是真正在使用工具，还是在“背书”。
*   **评估鲁棒性**：结合规则与双 LLM 评判的集成机制，减少了单一评判标准带来的偏见或死板。

### 8. 不足与局限
*   **单轮限制**：目前仅限于单轮对话任务，无法捕捉多轮对话中工具链调用、错误恢复或状态更新的复杂失败模式。
*   **领域覆盖**：虽然涵盖了五个领域，但尚未包含代码生成、科学文献检索或多步骤工作流等场景。
*   **评判员依赖**：尽管使用了集成法，但 LLM 评判员仍可能存在潜在的共同盲点。
*   **静态性**：随着模型更新，基准测试可能面临数据污染风险，未来需要持续更新任务变体。
