## 摘要
代码复用是软件开发项目中的普遍做法，这暗示了对所复用代码的内在信任。然而，目前尚缺乏对开发者信任的根本理解，以及各种因素如何塑造其基于信任的认知过程。借鉴服从心理学和信任心理学，我们展示了首个受控实验（n=37）的结果，该实验利用眼动追踪技术探索了紧迫感（以代码优先级表示）和声誉（以代码作者的经验水平表示）如何影响开发者对代码可信度的感知。我们的研究发现，分配给代码补丁的优先级显著影响了开发者的代码审查行为，进而影响了他们的评估时间、认知负荷和感知质量。然而，合并和实现代码的决策并未受到影响。眼动追踪数据显示，在标记为由资深开发者与初级开发者编写的相同代码补丁中，整体视觉代码扫描和注意力分布存在差异。然而，在性能表现上没有显著差异。此外，参与者将代码功能、质量和可理解性列为代码评估的主要因素。尽管代码审查行为发生了明显变化，但令人惊讶的是，参与者忽视了紧迫感和声誉对他们审查和复用代码更改决策的实质性影响。本研究为深入理解软件工程中的信任迈出了下一步，并可能为未来关于代码审查平台与指南、代码复用以及自动代码生成的研究提供参考。

## Abstract
Code reuse is a widespread practice across software development projects, suggesting an inherent trust in the reused code. Yet, there is a lack of a fundamental understanding of developers' trust and how various factors mold their trust-based cognitive processes. Drawing from the psychology of compliance and trust, we present the results of the first controlled experiment (n=37) which uses eye tracking to explore how urgency (represented by code priority level) and reputation (represented by the experience level of the code's author) influence developers' perceptions of code trustworthiness.   Our research revealed that the priority assigned to a code patch significantly influenced developers' code review behavior, impacting their evaluation time, cognitive load, and perceived quality. However, the decision to incorporate and implement the code was not affected . Eye tracking data revealed that there were variations in overall visual code scanning and the distribution of attention across identical code patches labeled as written by senior vs. junior developers. Yet, there were no significant performance differences. Moreover, our participants nominate code functionality, quality, and comprehensibility as primary factors in code evaluation.   Despite noticeable changes in code review behavior, our participants surprisingly overlooked the substantial influence of urgency and reputation on their decisions to review and reuse code changes. This study takes the next step toward a better understanding of trust in software engineering and may inform future research about code review platforms and guidelines, code reuse, and automated code generation.

### 论文总结：信任之眼——通过紧迫感与声誉探索开发者的信任感知

#### 1. 核心问题与整体含义（研究动机和背景）
代码复用在软件开发中极为普遍，其背后隐含着开发者对他人代码的“信任”。然而，软件工程领域对开发者如何形成这种信任、以及哪些非技术因素会干扰其认知过程缺乏深入理解。本研究旨在探讨**紧迫感（Urgency，通过代码优先级体现）**和**声誉（Reputation，通过作者经验水平体现）**如何影响开发者在代码审查过程中的信任感知、行为模式及最终决策。研究背景基于心理学中的“启发式-系统处理模型”（HSM），即人类在压力或面对特定线索时，往往会依赖直觉而非逻辑分析。

#### 2. 论文提出的方法论
研究采用了**受控实验（Controlled Experiment）结合眼动追踪（Eye Tracking）**的方法：
*   **核心思想**：通过操纵代码补丁的外部标签（优先级和作者背景），同时保持代码实际质量一致，观察开发者在真实开发环境（IDE）中的行为差异。
*   **关键技术细节**：
    *   **眼动追踪**：使用 Tobii Pro Fusion 硬件捕捉注视点（Fixations）和眼跳（Saccades），衡量认知负荷。
    *   **iTrace 插件**：在 Eclipse IDE 中集成眼动数据，使其能精确关联到源代码的具体行和区域（AOI）。
    *   **双路径认知模型**：分析开发者是进行“系统处理”（逻辑分析代码）还是“启发式处理”（依赖作者名声或紧急程度）。

#### 3. 实验设计
*   **数据集与场景**：选取了 Java 开源项目 **jFreeChart**（约 30 万行代码）中的 6 个历史真实 Bug 补丁。
*   **实验因子**：
    *   **优先级**：标记为“高（High）”或“低（Low）”。
    *   **作者经验**：标记为“资深（Senior, $100/hr）”或“初级（Junior, $40/hr）”。
*   **参与者**：37 名具备 Java 经验的开发者（主要是研究生）。
*   **Benchmark/对比**：实验采用内组设计（Within-subjects design），每个参与者随机处理不同标签组合的任务，对比其在不同条件下的表现。
*   **测量指标**：性能指标（准确率、时间）、眼动指标（注视次数、总注视时长、平均注视时间）以及主观信任量表（Likert Scale）。

#### 4. 资源与算力
*   **硬件资源**：实验使用了 Tobii Pro Fusion 眼动仪（采样率 250 Hz）和 27 英寸显示器（1920x1080）。
*   **算力说明**：本研究属于人类行为实验，**未涉及大规模 GPU 算力或深度学习模型训练**。数据处理主要涉及统计学分析（如 Wilcoxon 符号秩检验、Chi-squared 检验）。

#### 5. 实验数量与充分性
*   **实验规模**：共 37 名参与者，每人完成 6 个代码审查任务，总计产生了 **222 组任务数据**。
*   **充分性评价**：实验设计较为充分且严谨。通过随机化任务顺序消除了学习效应（Learning Effect），并利用 iTrace 解决了 IDE 中滚动和切换文件带来的眼动偏移问题。样本量在眼动追踪类软件工程研究中属于中上水平，具有较好的统计效力。

#### 6. 主要结论与发现
*   **紧迫感的影响**：高优先级补丁显著增加了开发者的**审查时间**和**认知负荷**（视觉努力），且开发者主观上认为高优先级补丁质量更高（即便代码完全相同）。但优先级并未改变最终的“接受/拒绝”决策。
*   **声誉的影响**：作者经验水平改变了开发者的**视觉扫描模式**（例如在审查资深作者代码时更关注相关方法），但在任务准确率、耗时或最终采纳决策上没有显著差异。
*   **认知脱节**：开发者在访谈中声称自己只关注功能、质量和可理解性，主观上认为优先级和声誉不影响决策。然而，眼动数据证明他们的**认知过程已被这些外部因素潜移默化地改变**。

#### 7. 优点（亮点）
*   **多模态数据结合**：将客观的眼动生理数据与主观的自我报告相结合，揭示了开发者“言行不一”的认知偏差。
*   **高生态效度**：实验在真实的 IDE 环境中进行，允许参与者滚动、编辑和运行测试，比以往使用静态代码片段的研究更贴近实际开发场景。
*   **控制变量严谨**：通过去除头像、使用缩写等方式排除了性别、年龄、外貌等潜在干扰因素。

#### 8. 不足与局限
*   **参与者构成**：参与者主要是研究生，虽然具备专业知识，但与拥有多年行业经验的职业开发者在信任倾向和审查习惯上可能存在差异。
*   **项目单一性**：仅使用了 jFreeChart 一个项目和 Java 语言，结论在其他领域（如嵌入式或前端）或编程语言中的普适性有待验证。
*   **任务时限**：每个任务限制在 10 分钟内，这可能人为制造了额外的压力，影响了自然状态下的信任评估。
*   **偏差风险**：虽然控制了代码质量，但未深入探讨不同类型的 Bug（如逻辑错误 vs. 语法错误）是否会与信任线索产生交互作用。
