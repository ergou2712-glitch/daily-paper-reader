## 摘要
基于大语言模型（LLM）的软件工程助手不仅会因为产生错误输出而失败，当代码、文档和测试不一致时，还会因为将信任分配给错误的制品而导致失败。现有的评估主要关注下游结果，因此无法揭示模型是否识别了退化的证据、确定了不可靠的来源，或者是否在不同制品之间校准了其信任度。我们提出了 TRACE（面向校准评估的制品信任推理），这是一个在盲扰动下，针对 Javadoc、方法签名、实现和测试前缀提取结构化制品级信任追踪的框架。利用来自 7 个模型在 456 个精选 Java 方法包上的 22,339 条有效追踪，我们评估了各制品的质量评估、不一致性检测、受影响制品归因以及来源优先级排序。在所有模型中，质量惩罚主要局限于受扰动的制品，并随严重程度增加，但敏感性在不同制品类型间是不对称的：文档错误引发的“严重-细微”差距（0.152-0.253）显著大于实现故障（0.049-0.123）。模型能很好地检测显式文档错误（67-94%）以及 Javadoc 与实现之间的矛盾（50-91%），但当仅实现发生偏移而文档保持合理时，模型表现出系统性的盲点，检测率下降了 7-42 个百分点。在 7 个模型中，有 6 个模型的置信度校准较差。这些发现表明，目前的 LLM 更擅长审计自然语言规范，而非检测细微的代码级偏移，这促使在对正确性要求极高的下游使用之前，进行显式的制品级信任推理。

## Abstract
LLM-based software engineering assistants fail not only by producing incorrect outputs, but also by allocating trust to the wrong artifact when code, documentation, and tests disagree. Existing evaluations focus mainly on downstream outcomes and therefore cannot reveal whether a model recognized degraded evidence, identified the unreliable source, or calibrated its trust across artifacts. We present TRACE (Trust Reasoning over Artifacts for Calibrated Evaluation), a framework that elicits structured artifact-level trust traces over Javadoc, method signatures, implementations, and test prefixes under blind perturbations. Using 22,339 valid traces from seven models on 456 curated Java method bundles, we evaluate per-artifact quality assessment, inconsistency detection, affected artifact attribution, and source prioritization. Across all models, quality penalties are largely localized to the perturbed artifact and increase with severity, but sensitivity is asymmetric across artifact types: documentation bugs induce a substantially larger heavy-to-subtle gap than implementation faults (0.152-0.253 vs. 0.049-0.123). Models detect explicit documentation bugs well (67-94%) and Javadoc and implementation contradictions at 50-91%, yet show a systematic blind spot when only the implementation drifts while the documentation remains plausible, with detection dropping by 7-42 percentage points. Confidence is poorly calibrated for six of seven models. These findings suggest that current LLMs are better at auditing natural-language specifications than at detecting subtle code-level drift, motivating explicit artifact-level trust reasoning before correctness-critical downstream use.

### 论文总结：衡量大语言模型在冲突软件制品间的信任分配

#### 1. 核心问题与整体含义（研究动机和背景）
在软件工程（SE）任务中，LLM 通常需要处理由源代码、方法签名、文档（Javadoc）和测试代码组成的“制品包”。然而，这些制品在现实中往往存在不完整、过时或相互矛盾的情况。现有的评估方法主要关注 LLM 输出的最终正确性（如下游代码生成的准确率），却忽略了模型在面对冲突证据时如何分配“信任”。如果模型盲目信任了错误的文档而忽略了正确的代码，这种“隐性失败”很难通过最终输出察觉。本研究旨在通过显式地测量模型对不同制品的信任分配，揭示模型在处理冲突信息时的推理过程和局限性。

#### 2. 方法论：TRACE 框架
研究者提出了 **TRACE (Trust Reasoning over Artifacts for Calibrated Evaluation)** 框架，其核心思想是将“制品级信任推理”作为模型的一等输出。
*   **核心组件**：TRACE 要求模型生成结构化的推理追踪（Reasoning Traces），包含：
    *   **每项制品质量评估 (A)**：对 Javadoc、签名、实现（MUT）和测试前缀分别打分（0-1）。
    *   **不一致性判断 (C)**：包括成对冲突分析、识别冲突点及受影响制品的归因。
    *   **来源优先级排序 (P)**：对四个制品的可靠性进行全排序并给出置信度。
*   **关键技术细节**：
    *   **制品对称性原则**：不预设代码或文档哪个更正确，要求模型通过交叉验证来赢得可靠性。
    *   **盲扰动协议**：在不告知模型输入是否被修改的情况下，对制品进行 6 种类型的扰动（如删除描述、注入逻辑漏洞、制造文档与代码矛盾），并分为“严重、普通、细微”三个等级。

#### 3. 实验设计
*   **数据集**：基于 OE25（包含 25 个真实 Java 系统），精选出 456 个高质量 Java 方法包。
*   **扰动变体**：每个样本生成 7 个版本（1 个原始 + 6 个扰动），总计 3,192 个评估实例。
*   **对比模型**：涵盖了 7 个主流模型：Claude 3 Opus/Sonnet/Haiku、GPT-4o、GPT-5.2 Chat、DeepSeek-V3.2-Speciale、Grok 4 Fast。
*   **基准 (Benchmark)**：以人工标注的扰动来源、故障类型和严重程度作为 Ground Truth。

#### 4. 资源与算力
*   **算力说明**：论文提到所有模型均通过 Microsoft Foundry/Azure 服务运行。
*   **训练/推理时长**：由于这是一项评估研究而非模型训练研究，文中未提及 GPU 型号、数量或训练时长。实验主要涉及大规模的 API 推理调用（共计 22,339 条有效追踪）。

#### 5. 实验数量与充分性
*   **实验规模**：共收集了 22,339 条结构化推理追踪，每条追踪包含多个维度的评分和文本分析。
*   **充分性**：实验设计非常充分。研究不仅对比了不同厂商的模型，还通过消融实验分析了扰动严重程度、扰动类型（文档 vs 代码）对模型判断的影响。
*   **客观性**：采用了盲测协议，确保模型评分反映的是其内在的质量判断而非对提示词的顺从。

#### 6 主要结论与发现
*   **敏感性不对称**：模型对文档（自然语言）的退化非常敏感，但对代码逻辑的细微偏移反应迟钝。文档错误的评分下降幅度（0.152-0.253）远超代码错误（0.049-0.123）。
*   **实现侧盲点**：当文档看起来很合理但代码实现发生偏移时，模型的检测率会大幅下降 21-43%。这表明 LLM 倾向于过度信任自然语言规范。
*   **置信度校准极差**：在 7 个模型中，有 6 个模型的置信度分数无法区分其判断的对错（仅 DeepSeek-V3.2 表现出较好的校准性）。
*   **语义理解驱动**：模型在细微矛盾下的表现差异与其语义代码理解能力高度相关。Sonnet 和 DeepSeek 在处理无明显词汇差异的逻辑冲突时更稳健。

#### 7. 优点
*   **视角新颖**：首次形式化了软件制品间的“信任分配”问题，从关注“结果正确”转向关注“推理可靠”。
*   **结构化评估**：TRACE 框架强制模型输出 JSON 格式的推理过程，使得原本模糊的“信任”变得可量化、可对比。
*   **高质量基准**：构建了一个带有详细元数据标注（严重程度、故障类别）的扰动基准数据集，对后续研究具有参考价值。

#### 8. 不足与局限
*   **语言局限性**：实验仅限于 Java 语言，尚未验证在 Python 或 C++ 等其他语言上的泛化性。
*   **扰动偏差**：虽然经过人工校验，但部分扰动是由 LLM 生成的，可能存在模型特有的模式，导致评估结果在不同模型间存在潜在偏差。
*   **现实复杂性**：现实中的软件冲突往往比实验中的受控扰动更复杂、更分散，TRACE 目前主要处理方法级别的局部冲突。
